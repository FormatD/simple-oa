# 系统架构设计

## 1. 整体架构概览

```
+------------------+     +-------------------+     +------------------+
|   Frontend SPA   | <-> |   API Gateway     | <-> |  Auth Service    |
|  (Vue 3 / React) |     |   (REST + WS)     |     +------------------+
+------------------+     |                   | <-> |  Task Service    |
         |               |  Nginx / Traefik  |     +------------------+
         |               |                   | <-> |   HR Service     |
    CDN / Static         +-------------------+     +------------------+
                                                  |  Wiki Service    |
                                                  +------------------+
                                                  |  Notification    |
                                                  |  Service         |
                                                  +------------------+
                                                  |  Report Service  |
                                                  +------------------+
                                                  | Workflow Engine  |
                                                  |  (审批流引擎)     |
                                                  +------------------+
                                                          |
                                                  +--------+--------+
                                                  |                  |
                                          +-------+-------+  +------+------+
                                          |  PostgreSQL   |  |    Redis    |
                                          |  (主数据库)    |  |  (缓存/队列)|
                                          +---------------+  +-------------+
                                                                |
                                                        +-------+--------+
                                                        | Object Storage |
                                                        | (文件附件)      |

```

## 2. 分层架构

### 2.1 前端层 (Frontend)

- 技术栈: Vue 3 + TypeScript / React 18 + TypeScript
- 状态管理: Pinia (Vue) / Zustand (React)
- 路由: Vue Router / React Router v6
- UI 框架: Ant Design / Element Plus
- 实时通信: WebSocket (Socket.IO)
- HTTP 客户端: Axios
- 编辑器: TipTap (富文本) + CodeMirror (Markdown)

### 2.2 API 网关层

- 职责: 路由转发、统一鉴权、限流、日志
- 方案: Nginx / Traefik
- 认证: JWT Token (Access + Refresh Token)

### 2.3 后端服务层

采用微服务架构，按业务域拆分：

| 服务名称 | 职责 | 技术建议 |
|----------|------|----------|
| Auth Service | 用户认证、角色权限、业务角色 | Node.js / Go |
| User Service | 用户管理、组织架构、岗位体系 | Node.js / Go |
| HR Service | 员工档案、考勤、假期、绩效、薪酬、招聘 | Node.js / Go |
| Task Service | 任务CRUD、状态流转、看板 | Node.js / Go |
| Wiki Service | 文档管理、版本控制 | Node.js / Go |
| Notification Service | 通知推送、消息队列 | Node.js / Go |
| Report Service | 统计报表、数据导出、数据导入 | Node.js / Go |
| Audit Service | 操作日志、审计追踪 | Node.js / Go |
| Workflow Engine | 审批流引擎（请假、报销、采购等审批统一驱动） | Node.js / Go / Java |

> 小型部署方案可合并为单体应用，模块间通过接口解耦，未来按需拆分。

### 2.4 数据层

- PostgreSQL: 所有结构化业务数据
- Redis: Session 缓存、任务队列 (Bull/BullMQ)、WebSocket 消息广播
- 对象存储: 文件附件 (S3/MinIO)

## 3. 认证与权限体系

### 3.1 认证流程

1. 用户登录 → 服务端验证凭证 → 颁发 JWT (Access Token: 1h, Refresh Token: 30d)
2. 客户端携带 Access Token 请求 API
3. API 网关校验 Token 有效性
4. Token 过期 → 使用 Refresh Token 自动续期

### 3.2 权限模型 (组织岗位权限模型)

权限与组织架构深度结合，通过"组织 -> 部门 -> 岗位 -> 用户"四级体系实现权限传递。

**核心设计**:
- 权限绑定到**岗位 (Position)**，而非直接绑定到用户或角色
- 用户通过担任特定岗位继承该岗位的所有权限
- 岗位在部门树中分层，形成清晰的权限等级
- 上级岗位自动继承下属岗位的权限（向上汇总）

**岗位层级**:
- CEO: 最高管理权限
- VP: 分管领域管理权限
- Director: 部门级管理权限
- Manager: 团队级管理权限，自动获得审批权
- Staff: 基础操作权限
- Intern: 受限操作权限

**权限继承规则**:
```
CEO                  → 所有权限
  └── VP            → 分管领域 + 以下所有
        └── Director → 部门级 + 以下所有
              └── Manager → 团队级 + 审批权
                    └── Staff → 基础操作
                          └── Intern → 只读受限
```

**权限粒度**:
- 全局权限: 系统管理、用户管理、组织配置（仅 CEO/VP）
- 部门级权限: 部门内资源操作（部门负责人自动获得）
- 资源级权限: 单个任务/文档的读写（岗位定义）

### 3.3 业务角色权限 (Business Role)

在岗位权限基础上叠加**业务角色**，用于跨部门的职能授权，解决岗位层级无法覆盖的水平授权场景。

**设计要点**:
- 业务角色独立于组织架构树，可跨部门/跨团队授予
- 业务角色不与岗位层级挂钩，平级岗位可拥有不同业务角色
- 业务角色与岗位权限**叠加生效**：用户同时拥有岗位来源和业务角色来源的权限并集
- 业务角色由管理员通过配置界面动态分配，无需修改代码

**典型业务角色**:
| 角色 | 授权范围 | 说明 |
|------|----------|------|
| 财务审批人 | 跨部门审批费用报销、采购申请 | 独立于岗位，可授予任何岗位用户 |
| HRBP | 跨部门查看员工信息、参与绩效评估 | 支持人力资源业务伙伴模式 |
| 项目经理 | 跨部门协调项目资源、查看项目任务 | 支持矩阵式项目管理 |
| 合规审核人 | 查看审计日志、合规检查 | 满足合规审计需求 |

**权限校验流程**:
```
用户请求 → 获取岗位权限 (Position ACL)
         → 获取业务角色权限 (Business Role)
         → 合并为完整权限集
         → 校验是否包含目标权限
         → 通过/拒绝
```

### 3.4 数据隔离

- 多租户模型: 每个组织一个独立租户
- 所有数据通过 organization_id 隔离
- 部门间数据通过 department_id + 岗位层级控制访问
- 上级岗位可查看下属所有数据
- 业务角色可跨越部门边界访问指定资源

## 4. 通用工作流引擎

### 4.1 定位与职责

工作流引擎独立于各业务服务，统一管理所有审批流程的定义和执行。它将审批逻辑从业务代码中抽离，使新审批需求的接入无需修改核心业务代码。

### 4.2 核心能力

- **流程定义**: 支持可视化 BPMN 或 DSL 方式定义审批流
- **节点类型**:
  - 顺序审批: 按指定顺序依次审批
  - 会签: 所有审批人通过即通过
  - 或签: 任一审批人通过即通过
  - 加签: 审批中可动态增加审批人
  - 转审: 审批人可将任务转给他人
- **条件分支**: 基于金额、岗位等级、部门等条件自动路由到不同审批链
- **超时处理**: 审批超时自动升级到上级或指定人
- **历史追溯**: 完整的审批轨迹记录

### 4.3 与业务服务的交互

```
业务服务 (HR/Task/etc.)
       │
       ▼ 调用工作流API
Workflow Engine
       │
       ▼ 执行流程定义
  审批节点 → 通知审批人 → 等待响应
       │
       ▼ 回调业务服务
  审批完成 → 业务服务处理后续逻辑
```

### 4.4 候选方案

| 方案 | 优势 | 劣势 |
|------|------|------|
| Flowable | 功能完备，BPMN 标准 | 较重，Java 栈 |
| Camunda | 轻量级，REST API 友好 | 同上 |
| 自研轻量引擎 | 与系统无缝集成，可控性强 | 需要较高开发投入 |

> 建议方案：初期使用自研轻量引擎配合简单配置化审批规则，后续根据业务复杂度考虑迁移至 Camunda/Flowable。

## 5. 实时通信架构

```
Client <--WebSocket--> API Gateway --> Notification Service --> Redis Pub/Sub
                                            |
                                      Push Notification
                                      (Web / Email / SMS)
```

- WebSocket 连接由 API 网关统一管理
- 通知事件通过 Redis Pub/Sub 广播
- 离线用户通过邮件/消息推送补发

## 6. HR 管理模块架构

HR 管理是本系统的核心业务模块，涵盖以下子域：

```
HR Service
  ├── Employee Management   (员工档案、合同)
  ├── Position Management   (岗位定义、编制管理)
  ├── Attendance            (打卡、考勤统计、加班)
  ├── Leave Management      (请假类型、申请审批、额度)
  ├── Performance           (绩效考核、OKR/KPI、360评估)      ← 后续迭代
  ├── Recruitment           (职位发布、候选人、面试)
  └── Payroll               (薪酬结构、薪资核算、社保)          ← 后续迭代
```

每个子域作为服务内部的一个模块，数据结构相互关联，但 API 路由统一在 HR Service 下暴露。

**实施优先级**:
| 优先级 | 模块 | 阶段 |
|--------|------|------|
| P0 | 员工档案、岗位管理 | 第一阶段 |
| P0 | 考勤管理、假期管理 | 第一阶段 |
| P1 | 招聘管理 | 第二阶段 |
| P2 | 绩效管理 | 后续迭代 |
| P2 | 薪酬管理 | 后续迭代 |
| P3 | 费用报销、培训、入职/离职流程、福利管理 | 后续迭代 |

## 7. 数据导入架构

为支持系统上线时的历史数据迁移，设计专门的数据导入通道：

```
用户上传 CSV/Excel
       │
       ▼
  上传 → 临时文件存储 (对象存储)
       │
       ▼
  解析 → 数据校验 (字段合法性、关联性)
       │
       ▼
  预览 → 错误行高亮，用户修正
       │
       ▼
  确认导入 → 批量写入 (事务包裹)
       │
       ▼
  导入日志 → 成功/失败记录
```

**技术要点**:
- 大文件采用分片上传
- 校验阶段不写入数据库，确保预览可撤销
- 批量写入使用 PostgreSQL 事务保障数据一致性
- 支持自定义字段映射（导入模板配置）

## 8. 技术选型理由

| 组件 | 选型 | 理由 |
|------|------|------|
| 数据库 | PostgreSQL | 强事务支持，复杂查询能力，JSONB 灵活存储 |
| 缓存 | Redis | 成熟稳定，Pub/Sub 支持实时推送 |
| 消息队列 | Redis (Bull) | 轻量级，与缓存共用基础设施 |
| 对象存储 | MinIO / S3 | 高可用，成本可控 |
| API 文档 | OpenAPI 3.0 | 业界标准，自动生成客户端 |
| 容器化 | Docker + Docker Compose | 开发/部署一致，简化运维 |
| 工作流引擎 | 自研轻量 / Camunda | 看阶段选型（见第4节） |

## 9. 安全设计

- 所有 API 强制 HTTPS
- JWT 签名密钥定期轮换
- 密码 bcrypt 加密存储
- CORS 白名单控制
- 请求速率限制 (Rate Limiting)
- SQL 注入防护 (参数化查询)
- XSS/CSRF 防护 (CSP headers, SameSite Cookie)
- 敏感操作二次确认 (删除/批量操作)
- 岗位权限 + 业务角色在服务端双重校验，前端仅做展示
